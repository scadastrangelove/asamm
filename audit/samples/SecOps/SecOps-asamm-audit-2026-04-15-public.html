§1 Executive summary

SecOps is a cybersecurity SecOps platform with a deeply embedded agentic component Onotolle (LangChain/LangGraph over self-hosted LLM) that manages issues and threats, plus an agentic development workflow (Claude Code + PM-assistant + MCP). The system is pre-production (owner-confirmed: local dev only). This window is the opportunity to fix architectural gaps before client data enters the system.

6 Critical, 6 High, 5 Medium findings. Two Critical findings directly implement the owner's stated mission-breakers:

  • F-1 — Comprehensive ACL bypass of all Onotolle tools. API endpoints enforce user.has_access_to_project(issue.project_id); use-cases don't. Onotolle tools call use-cases directly, bypassing ACL for read, write, create, and publish. Verified by reading both paths side-by-side.
  • F-2 — Credential leakage. Engine API key committed to backend git history ([ abc1234]), confirmed via git log -S. Yandex.Tracker OAuth token plaintext in three disk locations (non-git repo root; not git-leaked, but disk-leak + archive-leak via sync-config.sh).

With corrected evidence quality, maturity is 15×L0, 4×L1, 0×L2, 0×L3 — effectively the null baseline for an agentic-security programme.

Recommendation: freeze any staging/prod rollout of Onotolle until §2 Immediate Actions are complete. Estimated effort: ≈4 hours of emergency fixes + ≈3 person-days for full Horizon-0 hardening with tests.

§4 What works well (before gap analysis)

  1. API layer enforces ACL consistently. 13+ occurrences of user.has_access_to_project(...) across app/api/v1/finding.py. The pattern exists and is consistent — Onotolle is simply not wired into it.
  2. Self-hosted LLM (settings.py:168 → http://192.168.x.x:8005/v1). LLM prompts and responses do not leave customer infra. Engine is the exception.
  3. Query parser is structurally safe against SQL injection. app/query/grammar.py uses pyparsing → AST → Specification pattern → typed SQLAlchemy expressions. No text() or string concatenation with user input.
  4. Strong audit trail via Event / log_event. update_issue_property (line 308) logs prev_val/next_val; publish_issue (line 1021) logs issue update + threat creation atomically. Onotolle-created issues carry source="onotolle" (finding.py:508) → distinguishable from human-created.
  5. IssueEditor field whitelist. finding.py:58-73 forbids edits to project_id, project_code, created_by, published_by. Agent cannot "teleport" issues between projects.
  6. UserDep preloads user.acls (auth.py:71-72). The data needed for enforcement is already in the request context.
  7. Worktree scripts are git-disciplined. create-mr.sh:109 is plain git push -u (no --force, no --no-verify); commit-count guard prevents pushing empty branches; Draft-MR opt-in exists (though not default — see F-13).
  8. Issue model carries project_id (app/model/finding.py:182). The WHERE clause for a fix is possible with no migration.
  9. Binary files rejected from LLM context (base.py:57-58): UTF-8 decode failure → placeholder instead of raw bytes.
  10. Carthage is non-agentic. Verified via carthage/CLAUDE.md — a CRUD EASM inventory prototype with httpx sync, no LLM.

§6 Empirical findings

Critical

Critical F-1 Comprehensive ACL bypass of the entire Onotolle tool surface. empirical

API endpoints (app/api/v1/finding.py:191, 203, 477, 493, 465, 505) execute if not user.has_access_to_project(issue.project_id): raise AccessDenied immediately before calling each use-case. The use-cases themselves (app/use_cases/finding.py:183 create_manual_issue, :268 update_issue_property, :982 publish_issue, :1042 unpublish_issue) do not perform this check; user_id is only used in log_event(..., created_by=user_id).

Onotolle tools call the use-cases directly: IssueEditorTool._arun at tools/finding.py:296 → update_issue_property; tools/finding.py:278 → publish_issue; tools/finding.py:532 → create_manual_issue. IssueSearcherToolV2 receives no user_id at all (factory.py:159) and builds select(Issue) with no tenant filter (tools/finding.py:367-368). IssueReaderTool calls uow.issues.get_issue_by_key which is also unfiltered (repository/finding.py:75-80).

The Issue model does carry project_id: uuid.UUID (app/model/finding.py:182), so the required WHERE clause is possible without migration. The session context ctx_get_session is a plain AsyncSession factory (deps/session.py:21) — it applies no automatic tenant filter.

Directly implements the owner's non-negotiable: "доступ к данным одного клиента со стороны другого клиента".

Current blast radius is bounded by the fact that deployment is local dev only and only the owner has access to the DB. Any staging rollout converts this to an immediate incident.

§8 Self-modification & ethics (mandatory section)

  • Onotolle prompt is read-only at runtime E. factory.py:122 reads system_v1.prompt once at factory init; :179-180 appends memory.md only for the deep agent. No .write_text() / open(..., "w") anywhere in app/onotolle/.
  • Subagent prompts are string constants E (subagents.py:15-37). Can only be changed via VCS commit.
  • Redis checkpoints persist conversation state; no TTL, no cleanup policy documented. AI-04 risk: sessions can be restored after restart, potentially under a different user context if chat_id collides (F-6).
  • User-side MEMORY.md is edited by user and (by Claude Code design) by the agent. sync-config.sh copies it between machines with a sed rewrite of the user prefix. This file is an agent-self-modification surface, and there is no write-log.
  • Value constraints. system_v1.prompt:61-67 forbids disclosure of API keys, system commands, configuration, architecture, authentication mechanisms, user/password/token information. No instruction about tenant isolation (F-18), no instruction against fabricating CVEs, no instruction against escalating severity without evidence — these would all be L1 behavioral floors, absent today.

§12 Framework gaps observed in ASAMM v0.2

  1. Application-layer ACL bypass is implicit in AI-03 but worth an explicit sub-control: tools should not bypass checks that API endpoints perform. SecOps shows the full shape of this bug class.
  2. Dev-time agentic workflows (Claude Code + workflow-agent) fall under AD-04 but the framework is thin on specifics around autonomous commit/push/MR, shared mutable MEMORY.md, and per-skill "read-only vs write" claims. For teams shipping with agentic development the current guidance under-specifies.
  3. MCP supply chain — uvx xxx@latest is a common anti-pattern that belongs under AG-02 but could use a dedicated check.
  4. Shared mutable memory between user and agent (MEMORY.md) is poorly covered by AI-04; the write-log requirement should be explicit for dual-write surfaces.
  5. Pre-prod readiness gate. ASAMM assumes an operating system; for pre-prod work it would help to have a "readiness to accept client data" control that collects L1-floor across families before any rollout.

§13 Final verdict

Maturity: L0.2 (effectively pre-programme). The SecOps agentic surface is not ready for client data. Two architectural gaps — comprehensive ACL bypass (F-1) and absent kill switch (F-3) — directly violate the owner's non-negotiable, and two behavioural gaps (F-4, F-18) mean even the last-line prompt-level defence against cross-tenant action is missing.

The owner accepts residual risk personally (Phase 1 Q3). Within the rules of Agentic SAMM, that acceptance is valid only for behavioural-only constraints on non-mission-critical boundaries; it does not cover the non-negotiable itself. The correct framing is: until F-1, F-2, F-3 are closed and F-18 has at least a behavioural floor, any deployment beyond the owner's single-user laptop is outside the accepted-risk envelope. This should be recorded in writing with a date.

Recommended sequence: Horizon 0 (§11) → Phase 4.5 independent verification (§0.6) → Horizon 1 → staging rollout with limited pilot → Horizon 2 in parallel.
