Recommendations

Ordered by ASAMM §2.3 greenfield priority · P1=containment · P2=observability · P3=verification
1
Conduct the mission interview (ASAMM §2.7 hard gate) P1 — process
Before acting on any finding: establish with Anton Razzhigaev what he cannot afford to lose, what counts as an irreversible action in this system, and what the current deployment scope actually is. A single-researcher research experiment at 48 hours old has a very different risk tolerance to a shared system. This interview determines whether Critical ratings are actionable immediately or deferred. It is a §2.7 hard requirement, not a recommendation.
2
Add a command allowlist to shell.py P1 — containment · AI-02
Implement an explicit command allowlist in shell.py (e.g., git, python, pip, curl to allowed hosts). Reject commands not on the list with a structured error logged to events.jsonl. This is the highest-impact single code change: it limits the blast radius of any successful injection to what is explicitly permitted. The desktop fork's "deterministic whitelist for known-safe ops" is the reference implementation. Verify the source first to determine whether any filtering already exists before implementing.
3
Block write-path operations during non-/evolve tasks P1 — containment · AI-02, AI-03
During read-only or analysis tasks (not /evolve), disable git push, git commit, and destructive shell operations at the tool invocation level — not just by omitting schemas from context. Use a read_only flag in task metadata that the tool layer checks before execution. This reduces the blast radius of the most common task type to near-zero for worst-case tool misuse, without changing /evolve behaviour.
4
Add a tool registry manifest with governance gate P1 — containment · AG-02
Create tools/REGISTRY.json with one entry per tool (name, risk_tier: always/conditional, blast_radius: low/medium/high). Change auto-discovery to load only tools present in the manifest. The agent cannot add a new tool without first adding a manifest entry — which requires a human commit. This closes the "tool surface expansion within an autonomy window" finding entirely and takes approximately 30 minutes to implement.
5
Add trust-level fields to events.jsonl P2 — observability · AO-01
Add two fields to every tool-call event: context_source (telegram / github_issue / web_search / drive / llm_response) and trust_level (owner / external / untrusted). This transforms events.jsonl from an operational log into a usable security audit trail — and is the prerequisite for any intent–action gap monitoring. One session to implement; permanent improvement to assurance visibility.
6
Implement plan-vs-invocation gap detection P2 — observability · AO-02
At task start, require the agent to declare its plan (intended tools and data sources). At task end, diff the declared plan against actual tool calls in events.jsonl. Any undeclared high-blast-radius invocation generates a Telegram alert to the owner. This would have caught all three documented gap events proactively. Bridges AO-02 from L0 to L1, and creates the data needed for L2 automated monitoring.
7
Add behavioral test cases for primary threat classes P3 — verification · AV-01
Add at minimum: one test with a mock GitHub issue containing an injection payload (verify no shell invocation occurs); one test with a web search result containing injection (verify no git push occurs); one test triggering the deduplication logic (regression for the $500 incident). These satisfy AV-01 L1. Separately, reclassify test_constitution.py in documentation as what it is: a specification record, not an integration test. Do not count it toward behavioral coverage metrics.
8
Define spiral reassessment triggers in SYSTEM.md P3 — process · AO-03
Add a mandatory pause section to SYSTEM.md: "The following events require an owner review checkpoint before resuming autonomous operation: adding a new tool, adding a new external context source, expanding MAX_ROUNDS, adding a new API connector." Agent pauses /evolve, sends a summary to Telegram, waits for /approve. This converts the 32-cycle unreviewed evolution history into a model with explicit security checkpoints at each capability boundary. AO-03 L1 requires only a defined trigger list — this satisfies it.
