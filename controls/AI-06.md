## Rationale

AI-03 governs what tools an agent can invoke per task. AI-06 governs the credentials and identities that make those invocations possible: how tokens, keys, service accounts, delegated permissions, impersonation grants, and agent identities are issued, scoped, rotated, attested, and revoked.

In multi-agent or multi-tool systems, credential delegation is the primary privilege escalation path. A compromised agent inherits every credential it holds, and a spawned subagent can become more dangerous than its parent if credential scope is copied rather than intentionally delegated.

Traditional non-human identity (NHI) controls answer whether a credential is valid. Agent identity must answer a harder runtime question: which agent is acting, who or what delegated the authority, what task or intent is the authority bound to, and whether that authority is still valid for the current action. A static service account may authenticate an agent, but it does not by itself provide accountability for agent reasoning, delegation, or tool use.

Agent identities should therefore be treated as governed, auditable principals rather than reused machine accounts. Where the platform supports it, credentials should be short-lived, task-scoped, and bound to provenance claims such as originating user, agent ID, session/task ID, trust grade, and delegated authority.

## Controls

| Level | Implementation |
|---|---|
| L1 | Credentials and agent identities available to each agent are inventoried; credential scope (which systems, what permissions, which authentication primitive) is documented per agent; credentials are not shared across agents with different trust grades |
| L2 | Credentials are scoped to minimum required permissions per agent per task; token lifetime is bounded (short-lived preferred); rotation and offboarding procedures exist and are tested; delegation chains preserve the originating user/agent/task context |
| L3 | Credential issuance and revocation are automated; agent identity is attested before sensitive credential issuance where supported; delegation chains are logged with full provenance; anomalous credential usage (scope expansion, off-hours access, cross-agent token reuse, stale spawned-agent identity) triggers alerts |

## Evidence criteria

| Level | Required evidence |
|---|---|
| L1 | Credential and agent identity inventory per agent; documented scope for each credential; evidence that agents with different trust grades do not share the same credential material |
| L2 | Least-privilege review record; tested rotation/offboarding procedure; token lifetime policy; delegation chain records preserving originating user/agent/task context |
| L3 | Automated issuance/revocation workflow demonstrated; attestation or policy decision record for sensitive credential issuance; provenance logs show credential delegation; alerting exists for anomalous credential usage |

## Scope note

For single-agent systems using a single API key, L1 may be satisfied by documenting that key's scope and confirming it does not grant permissions beyond the agent's operational need.

The control becomes critical when multiple agents share infrastructure, agents can delegate to subagents, agents hold credentials to external systems (client APIs, cloud services, repositories), or an agent can mint or exchange credentials during execution.

## NHI vs. agent identity

Use this distinction during audit:

| Question | Traditional NHI | Agent identity |
|---|---|---|
| What is verified? | Credential or service account validity | Acting agent, delegation context, task/intent, and current authority |
| Typical credential | API key, service account, OAuth client, certificate | Short-lived token, verifiable credential, attested runtime identity, delegated claim |
| Common failure | long-lived secret, overprivilege, reuse | trust transitivity, confused deputy, unbounded delegation, ghost agent |
| Evidence needed | credential inventory and rotation | identity chain, task binding, provenance logs, revocation path |

Agent identity can be implemented with different technical primitives. ASAMM does not require a specific mechanism such as SPIFFE, OIDC, DID, or OAuth token exchange. It requires that the system can demonstrate who acted, under whose authority, for which task, with which scope, and how that authority expires or is revoked.

## Offboarding and ghost agents

Agent-spawning architectures create short-lived principals. If those identities or credentials survive the workflow, they become dormant access paths. At L2 and above, offboarding must cover spawned agents, delegated credentials, temporary tokens, queued tasks, scheduled automations, and protocol peers. Evidence should show that credentials expire or are revoked when the workflow ends, not only when a human remembers to clean them up.
