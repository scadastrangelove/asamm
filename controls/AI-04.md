## Rationale

Any agent capability to write data that influences its own future behavior — persistent memory, scratch files, project instructions, system prompt extensions — creates a self-modification surface with cross-session blast radius. This surface is not covered by tool registry (AG-02) or execution boundary controls (AI-02).

A wrong architectural assumption written to cross-session memory by an agent silently biases all future development sessions. No external attacker is required: the blast radius originates from an incorrect write that persists without the user being notified or the write being logged. When platform properties change (vendor update removes or modifies the memory tool), structural controls that implicitly bounded this surface can disappear without warning.

**Self-modification surfaces that the user cannot audit must be treated as L0 regardless of agent self-report.**

## Controls

| Level | Implementation |
|---|---|
| L1 | All self-modification surfaces enumerated with cross-session flag; user-auditable mechanism exists for each (view command, UI panel, or API export); enumeration reviewed at least once per quarter |
| L2 | Writes logged with provenance (session identifier, tool call, what was written, when); high-blast-radius writes (architectural assumptions, operational rules, constraint inventory) require explicit user checkpoint before taking effect; user receives notification for any autonomous write to persistent state |
| L3 | Automated anomaly detection on self-modification patterns; unrecognized writes generate alerts; periodic automated review of persistent state against ground truth artifacts |

## Evidence criteria

| Level | Required evidence |
|---|---|
| L1 | Enumeration document listing each surface with cross-session flag; demonstration of user-auditable mechanism (view command output or UI screenshot) |
| L2 | Write log with provenance fields populated; checkpoint configuration or approval workflow for high-blast writes; notification mechanism demonstrated |
| L3 | Anomaly detection configuration; automated review schedule and output; alert samples |

## Evidence ceiling for self-report

All claims about what an agent "will not write" to persistent memory are [inferred] by default. L2 requires external logging or a user-auditable mechanism — not agent self-report. The only way to confirm current persistent state is to retrieve it via a user-auditable mechanism (e.g., `memory_user_edits view` in claude.ai), not to ask the agent to describe what it has written.

**Evidence quality note:** `[empirical absence]` — a view command that returns an empty or expected result — is stronger than `[unknown]`. It upgrades L1 from "we assume nothing was written" to "we checked and nothing unexpected was found." Quarterly review procedures should always produce `[empirical absence]` evidence, not `[unknown]`.

## Implementation notes

**Surfaces to enumerate (non-exhaustive):**
- Persistent memory (e.g., `memory_user_edits` in claude.ai, bio memory in ChatGPT, `~/.claude/projects/*/memory/` in Claude Code)
- Project-level instructions (e.g., `CLAUDE.md`, custom instructions)
- UI-mediated write surfaces: canvas documents, automations, scheduled tasks, integration state (Gmail drafts, Calendar events)
- Session-local scratch (ephemeral — note that this does NOT cross sessions)

**Automations as delayed execution surface (capability-partitioned environments):**
Automations are distinct from memory writes. An automation is created in one session and executes in future sessions without per-run confirmation. The blast radius is not "wrong state stored" but "wrong action executed later." Treat as SM-01 finding if agent-created automations exist without explicit user authorization per automation.

**Quarterly review procedure:**
1. Run the user-auditable mechanism (view command or UI panel)
2. For each product-related entry: verify against current codebase or ground truth
3. Delete or correct any entry that does not match current reality
4. Document: date, entries reviewed, corrections made
