**AI-02 — Execution Boundary Control**

SAMM function: Implementation | Threat coverage: C2, C3 | Path: B

| Level | Implementation |
|---|---|
| L1 | All agent-executed tool calls run in a defined execution boundary; filesystem and network access is documented per tool |
| L2 | Default execution policy is isolation; host execution requires explicit justification and approval; sandbox policy is enforced, not merely described |
| L3 | Sandbox policy violations are logged and alerted; escape paths are tested in adversarial evaluation; development-time tooling is sandboxed separately from production |

Evidence: sandbox policy document exists and matches deployed configuration; network and filesystem access has been verified under actual tool behavior, not only architecture review; escape attempt tests have been run.

## v0.2 extension — Platform Safety vs Workflow Safety

*Added: ASAMM v0.2*

**AI-02 grades execution boundary controls — this is platform safety only.**

Platform safety (sandbox strength, network controls, container isolation) is orthogonal to workflow safety (what data enters the pipeline, what advice is acted on without verification, what downstream systems consume the agent's outputs with what trust level). A high AI-02 grade does not compensate for low workflow safety.

**Definitions:**
- *Platform safety:* what the environment technically prevents. Primarily vendor-controlled for cloud-hosted agents. Assessed via empirical testing and vendor attestation.
- *Workflow safety:* what the actual usage pattern risks. User-controlled. Assessed via mission interview, policy review, and pipeline analysis.

**Required separation in audit reports:**
Any AI-02 grade must be accompanied by a separate workflow safety assessment. The two must never be merged into a single grade.

Example: an agent running in a gVisor sandbox with private IP blocking (AI-02: L2-vendor) can still have L0 workflow safety if scan output from hostile targets is uploaded without sanitization policy, outputs flow into a privileged local agent without trust marking, and no data classification governs what is permissible to upload.

**Shared responsibility for cloud-hosted environments:**
Platform safety for cloud-hosted agents should be labeled "vendor-attested" rather than L2, because the evidence base is vendor documentation and attestation (SOC 2, published architecture), not direct empirical testing of the underlying sandbox. The user-auditable portion of AI-02 is the workflow safety dimension.

**L2 addition:**
> Platform safety and workflow safety are graded and reported separately; workflow safety assessment includes data classification, pipeline trust mapping, and prohibited input policy.
