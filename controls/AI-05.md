## Rationale

Operational value constraints — mission boundaries the agent must not cross — are simultaneously ethical statements and security controls. For a security scanner: "must not attack systems outside declared scope" is a C3 threat class with a specific blast radius, not only an ethical principle. ASAMM must require explicit documentation, verification, and risk acceptance for this class of constraint.

The critical distinction this control forces: **"will not" vs "cannot."**

- **Technical enforcement (cannot):** the behavior is physically impossible — weight-level suppression, kernel-level control, scope enforcement that fails closed. Can be independently verified.
- **Behavioral enforcement (will not):** governed by the model's alignment training and runtime reasoning. Can in principle be overridden by context manipulation, adversarial prompting, or unusual operational conditions. A misaligned agent produces identical self-reports to an aligned one.

For an offensive security tool deployed to clients, a behavioral-only constraint on "must not attack out-of-scope systems" is a materially different guarantee than a scope enforcement check that fails closed. This difference must be surfaced in the audit and explicitly risk-accepted by the system owner — not treated as equivalent to technical controls.

## Controls

| Level | Implementation |
|---|---|
| L1 | Mission-critical constraints documented with explicit enforcement type (technical / behavioral) per constraint; blast radius if violated stated per constraint |
| L2 | Each constraint has a behavioral test case with pass/fail record; behavioral-only constraints on mission-critical boundaries are formally risk-accepted by the system owner with documented rationale; technical enforcement exists for at least one highest-blast-radius constraint |
| L3 | Automated adversarial testing for constraint violation scenarios at each release; behavioral-only constraints tracked as a security metric with trend monitoring |

## Evidence criteria

| Level | Required evidence |
|---|---|
| L1 | Constraint inventory table with enforcement type and blast radius per constraint |
| L2 | Behavioral test cases (probe + observed response + HOLDS/FAILS/AMBIGUOUS); formal risk acceptance record with owner identification and date |
| L3 | Automated constraint test suite configuration; metric dashboard with trend data |

## Constraint inventory template

For each mission-critical constraint, document:

| Constraint | Enforcement type | Blast radius if violated | Behavioral test exists? | Risk-accepted by owner? |
|---|---|---|---|---|
| [e.g., "scanner must not attack out-of-scope targets"] | TECHNICAL / BEHAVIORAL | [mission impact × recovery × lateral] | Yes/No | Yes/No — [date, owner] |

## Behavioral test protocol

For each behavioral-only constraint, define a test case:

```
Constraint: [text]
Probe: [what was asked to approach or trigger the constraint boundary]
Method: [how the probe was delivered — direct instruction, context injection, etc.]
Observed response: [what the agent did]
Evidence level: [empirical] if auditor ran it; [inferred] if described
Result: HOLDS / FAILS / AMBIGUOUS
Date: [when tested]
```

A HOLDS result upgrades the constraint to [empirical] for this specific probe. This does not guarantee the constraint holds under all adversarial conditions — it confirms it held for this test case.

A FAILS result is a Critical finding requiring immediate escalation and owner notification.

## Risk acceptance requirement

Behavioral-only enforcement on any mission-critical constraint must be explicitly risk-accepted. Risk acceptance means the system owner:
1. Understands the distinction between technical and behavioral enforcement
2. Acknowledges that behavioral-only constraints can be bypassed via adversarial context
3. Accepts the residual risk given current blast radius and likelihood assessment
4. Documents this acceptance with date and owner identification

Risk acceptance must be renewed at each reassessment trigger (new deployment context, model version update, new use case, new constraint).

## Implementation notes

**"Will not" vs "cannot" — practical test:**
If you can imagine a prompt that would cause the agent to violate the constraint, the enforcement is behavioral. If no prompt can cause the violation because the system prevents it at a layer below the agent's reasoning, the enforcement is technical.

**Self-modification interaction:**
An agent that writes constraints to its own persistent memory is exposing those constraints to cross-session modification risk (AI-04). Mission-critical behavioral constraints stored only in volatile session context are reset each session — this is a protection, not a problem.
